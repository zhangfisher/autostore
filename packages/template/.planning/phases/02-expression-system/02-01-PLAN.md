<objective>
实现 Phase 02 表达式系统的第一层：基于**参数解构**的安全表达式解析器（ExpressionParser）与简化后的 Binding 类。

Purpose: CONTEXT.md 已更新（D-37~D-39）——表达式求值采用**参数解构**策略：`ExpressionParser.parse("order.count")` 返回 `(scope) => (({...scope}) => order.count)(scope)`。这样表达式字符串 `order.count` 在被 `new Function` 求值时，能直接访问从 scope 解构出的 `order` 变量，无需手工逐层路径查找。AutoStore 的 computed 系统在 getter 内对 scope 的读取会自动追踪依赖（D-29），因此参数解构模式天然兼容 AutoStore 的依赖收集机制。

简化设计（D-11）移除 ExpressionBridge 中间层，Binding 直接负责：(1) 根据 directiveName 从 DirectiveRegistry 查找指令对象（D-07）；(2) 用 ExpressionParser 把表达式字符串安全解析为求值函数（EXPR-01/02/03 + D-04 + D-37~D-39）；(3) 调用 `store.computedObjects.create()` 创建计算对象（D-02/D-08/D-29/D-30）；(4) 监听计算对象变化并触发 `directive.update(el, value)`（D-09/D-10）。

**安全模型关键点：** 参数解构使用 `new Function` 求值表达式字符串，因此**安全校验必须前置**——在调用 `new Function` 之前先用正则黑名单 reject 所有危险语法（函数调用 `()`、赋值 `=`、`new`、`=>`、`;`、`[]` 等）。校验通过后才进入 `new Function` 包装。字面量表达式走单独分支，不进入 `new Function`（避免无谓开销，且字面量无依赖）。

Output: ExpressionParser.ts、Binding.ts、types.ts 扩展、以及两组单元测试。

       - 方法 `parse(expr: string, options?: ExpressionOptions): ParsedExpression | null`：返回 { dependencies, evaluate }。
       - 方法 `getDependencies(expr: string): string[]`：提取标识符路径（静态分析）。
       - 方法 `parseAndEval(expr: string, scope: any, options?: ExpressionOptions): any`：便捷方法，解析并立即求值。

       实现策略（D-37~D-39 参数解构）：

       a. **空表达式校验**：parse 入口先检查 `expr.trim() === ""`，空则抛 ExpressionParseError（或按 options.throws 返回 null）。

       b. **字面量分支**（EXPR-02）：用正则匹配字面量——
          - 单引号字符串：`^'([^']*)'$` → evaluate 返回捕获组内容，dependencies=[]。
          - 双引号字符串：`^"([^"]*)"$` → 同上。
          - 纯数字：`^-?\d+(\.\d+)?$` → evaluate 返回 Number(expr)，dependencies=[]。
          - 布尔：`^(true|false)$` → evaluate 返回 expr === "true"，dependencies=[]。
          字面量直接返回，**不进入 new Function**（避免无谓开销，且字面量无依赖）。

       c. **安全校验**（EXPR-03，在 new Function 之前执行）：对非字面量表达式跑 rejectUnsafe(expr)，用正则黑名单检测：
          - `/\bfunction\b/`（function 关键字）
          - `/=>/`（箭头函数）
          - `/\bnew\s+/`（构造调用）
          - `/;/`（多语句）
          - `/[\(\)]/`（函数调用语法，含 evil()、alert(1)）
          - `/[\[\]]/`（索引访问，v1 限制）
          - 赋值 `/(?<![=!<>])=(?![=])/`（排除 == === != !== <= >=）
          命中任一即抛 ExpressionParseError（或按 options.throws 返回 null）。
          **这是参数解构模式的安全护栏**——new Function 会执行表达式字符串，必须先拒绝所有危险语法。

       d. **参数解构求值**（D-37/D-38/D-39，核心实现）：对通过安全校验的标识符路径表达式，使用 `new Function` 生成参数解构求值函数：
          ```
          const fn = new Function('scope', `
            return (({...scope}) => {
              return (${expr});
            })(scope);
          `);
          ```
          生成的函数等效于 `(scope) => (({...scope}) => order.count)(scope)`：
          - 外层接收 scope（AutoStore 根状态响应式代理）
          - 内层用 `{...scope}` 解构参数，使表达式中的 `order`、`user` 等标识符可直接访问
          - 表达式字符串作为返回值插入，由 JS 引擎在解构后的环境中求值
          evaluate(scope) 即调用此 fn。

       e. **依赖提取**（getDependencies）：对标识符路径表达式，用正则 `/^[A-Za-z_$][\w$]*(?:\.[A-Za-z_$][\w$]*)*$/` 校验整串仅为点号路径，合法则返回 `[expr]`（整串作为单一完整依赖路径，如 "user.name" → ["user.name"]）。非纯路径（但通过安全校验的复合表达式，如 v1 暂不支持的算术）返回从表达式中提取的所有顶层标识符路径。**注意：依赖以静态分析结果为准，不依赖运行时解构展开的 get 事件**——因为 `{...scope}` 会展开整个 scope，若让 AutoStore 自动追踪会过度收集依赖。因此 parse 返回的 ParsedExpression.dependencies 必须由静态分析得出，Binding 在创建计算对象时通过 options.dependencies 显式声明（见 Task 2 action 第 2.c 步）。

       f. **容错选项**（D-22~D-24）：options.throws === false 时，parse 捕获所有错误（安全校验失败、空表达式、new Function 语法错误）返回 null（不抛出），由调用方（Binding）决定是否 warn。默认 throws=true 保持原抛出语义。
